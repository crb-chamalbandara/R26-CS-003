Problem, objective and novelty — from your slides
Exactly what your proposal states
▼
Problem (from slide 10)

Malicious browser extensions can pass store checks and get installed without raising any flags. After installation, they secretly steal data or track user activity. Traditional tools only scan the code statically and do not monitor real runtime behaviour — so hidden malicious actions go completely undetected.

Research objective

To develop a detection mechanism that identifies both known and unknown malicious browser extensions by combining static analysis and dynamic analysis into a single unified pipeline.

Novelty (your key claim)
Unlike existing tools (CRXcavator, Tarnish) that only detect known threats using signatures and permission checks, Component 1 combines ML-based classification with live sandbox behaviour monitoring to detect malicious extensions — including brand new, never-seen-before ones — based on what they actually do at runtime.
M
Your three implementation modules — from WBS slide
Static Analysis, Dynamic Sandbox, Verdict Engine
▼
1
Static Analysis Module — checks extension permissions and code structure, uses the XGBoost ML model, and outputs a static threat score. Covers: hash + signature detection, ML classification of manifest features, zero-day extension detection through permission patterns.
2
Dynamic Analysis Module (Sandbox) — only triggered when static score is above threshold. Runs the extension inside a Puppeteer headless browser isolated inside Docker. Monitors: sandbox execution, behaviour profiling, Chrome API call logging, outbound network request capture.
3
Verdict Engine — combines static score and dynamic behaviour findings into one final threat score, a list of specific alert flags, and a human-readable report. Saves to database and feeds the shared Central Dashboard.

Evaluation metrics — from slide 12
How you will measure your component's success
▼

The system is evaluated using precision, recall, and F1-score, comparing the full pipeline against static-only and dynamic-only approaches. False positives are measured using popular benign extensions, and the time taken from extension submission to final verdict is also recorded.

Precision

Of all extensions flagged as malicious, what percentage were actually malicious? Measures false alarm rate.

Recall

Of all actually malicious extensions, what percentage did the system catch? Measures miss rate.

F1-score

Harmonic mean of precision and recall. The single most important metric for imbalanced datasets.

Verdict time

How fast does the system go from receiving the extension file to outputting a verdict? Must be practical.

Your confirmed datasets — from slide 13 of your presentation
Malicious class Free Dataset 1 of 4
chrome-mal-ids — Malicious Extension ID Blocklist
github.com/mallorybowes/chrome-mal-ids
A continuously updated GitHub repository maintained by security researcher Mallory Bowes. Contains a blocklist of known malicious Chrome Extension IDs — extensions confirmed as adware, spyware, data stealers, or policy violators.
Contains a README.md and a malicious-extensions.txt or similar list file with one Extension ID per line. Each entry is a 32-character alphanumeric string (e.g., "hdokiejnpimakedhajhdlcegeplioahd").
How to use it: Take each Extension ID → query Chrome Web Store for its manifest.json → extract permissions → label as malicious (1) in your training CSV. Also use the IDs to build your hash database by downloading the .crx file for each.
How to download
git clone https://github.com/mallorybowes/chrome-mal-ids.git
Malicious class Free Dataset 2 of 4
palant/malicious-extensions-list — Curated Malicious Extensions
github.com/palant/malicious-extensions-list
Maintained by Wladimir Palant, the creator of Adblock Plus. Contains a detailed, manually curated list of malicious Chrome and Firefox extensions with descriptions of what each one does — spyware, search hijacker, credential thief, etc.
Higher quality than a raw blocklist because each entry explains the malicious behaviour. This is valuable for your sandbox testing — you know exactly what behaviour to expect from each extension, so you can verify your sandbox catches it.
How to use it: Use the Extension IDs to collect manifest data for your training set. Also use the behaviour descriptions to design your sandbox detection rules — e.g., if a listed extension is a "session cookie stealer", confirm your sandbox catches cookie access from that extension.
git clone https://github.com/palant/malicious-extensions-list.git
Malicious class Free Dataset 3 of 4
Chrome-Stats Malware Corpus — Removed Extensions
chrome-stats.com/search?obsoleteReason=malware
Chrome-Stats tracks every extension removed from the Chrome Web Store and the reason for removal. Filtering by obsoleteReason=malware gives you extensions confirmed removed by Google specifically due to malware classification — the highest quality malicious label you can get.
The website lists Extension ID, name, removal date, install count at removal, and sometimes permission details. These are real-world confirmed malicious extensions that passed store checks before being caught — exactly the type your component needs to detect.
How to collect: Visit the URL, scrape or manually export the Extension IDs. For each ID, retrieve the archived manifest.json from cached sources (Wayback Machine archives or CRXcavator historical data). Add to your malicious training set.
Note
These extensions are no longer live on the Web Store. To get their manifest data, use the CRXcavator API (which archives extension data) or search the Internet Archive for cached versions. The Extension ID is still usable as a hash lookup key if you can find an archived .crx file.
Malicious class Free Dataset 4 of 4 — most important for ML
refade/GoogleChromeExtension — 990 Malicious CRX + 1,098 Features
github.com/refade/GoogleChromeExtension
This is your most immediately useful dataset for ML model training. It contains 990 confirmed malicious Chrome extensions with 1,098 pre-extracted features per extension — permissions, manifest metadata, code structure signals, and more. The feature extraction is already done, which means you can start ML training almost immediately.
The dataset comes from an academic research paper on malicious extension detection. It is structured as a CSV or similar format ready for direct use with scikit-learn and XGBoost. This is the closest thing to a plug-and-play dataset for your exact use case.
How to use it: Download the repository. Load the feature CSV into pandas. This becomes your malicious class (label=1). Combine with benign extension features (label=0) to create the full training dataset. Train XGBoost directly on this combined CSV.
git clone https://github.com/refade/GoogleChromeExtension.git
pip install pandas scikit-learn xgboost

Top Chrome Web Store Extensions — Safe Baseline
chrome.google.com/webstore + CRXcavator API
The ML model cannot learn without seeing both malicious AND benign examples. Your benign class must be collected from the top-rated, high-install-count Chrome extensions that have been live for 1+ years without removal. Target 500–1000 extensions.
What to collect per extension: Extension ID, all declared permissions (as binary features — 1 if requested, 0 if not), number of host permissions, content script targets, presence of background script, total permission count. Label = 0 (benign).
Best source: CRXcavator API (crxcavator.io/api/v1) — query by Extension ID and get full metadata including all permissions. Also query the Chrome Web Store API for top extensions by category. Focus on productivity, developer tools, and accessibility categories which are dominated by legitimate extensions.
