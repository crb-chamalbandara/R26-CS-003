The core purpose of this dataset
Why it was built and what problem it solves
▼

This dataset was built to address a fundamental problem in cybersecurity: commercial antivirus products are bad at detecting malicious browser extensions. The researchers showed that even the best antivirus (ESET-NOD32) only detected 54.55% of the malicious extensions tested — meaning nearly half slipped through undetected.

The dataset provides a collection of Chrome extension files (.crx format) — both malicious and benign — that have been run inside a controlled sandbox environment to capture exactly what each extension does when it executes. This behavioural data is then converted into 1,098 numerical features that a machine learning model can learn from.

The key idea
Instead of looking at what an extension says it does (static analysis), this dataset captures what the extension actually does at runtime (dynamic analysis). This is why it is directly relevant to Component 1's sandbox analysis stage.

There is a major imbalance — only 22 malicious samples compared to 990 benign ones. A naïve ML model trained on this would simply learn to always predict "benign" and be right 97.8% of the time — which is useless for detection.

How the researchers solved this — biomedical-inspired strategy
They borrowed a technique from medical diagnostics where disease cases are rare. The training is repeated 45 times (ratio 990÷22 ≈ 45). At each iteration, all 22 malicious samples are paired with a fresh random batch of 22 benign samples. This ensures the model sees a balanced 22:22 split every iteration while still being exposed to the full diversity of benign extensions.

For your Component 1

You should apply the same technique — or use XGBoost's built-in scale_pos_weight parameter which automatically handles class imbalance.

Practical note

When combining this with other datasets (chrome-mal-ids, palant list), your malicious class will grow — reducing the imbalance problem significantly.

The 1,098 features — what they measure
🔬
Dynamic feature extraction — Cuckoo Sandbox
How the 1,098 features are generated from each extension file
▼

Each extension (.crx file) is executed inside Cuckoo Sandbox — an open source automated malware analysis system. While the extension runs, 1,098 behavioural signatures are monitored and logged. Each feature is a binary flag (0 or 1) representing whether a specific suspicious behaviour was observed during execution.

Why 1,098 features and not just permissions?
Static permission analysis (what the manifest declares) only tells you what the extension asks for. Dynamic features tell you what it actually does. An extension can declare benign permissions and still perform malicious actions at runtime — the 1,098 features capture this hidden behaviour.
Feature categories
Virtual machine detection — Does the extension try to detect if it is running inside Sandboxie, VirtualBox, VMware, or other analysis environments? Malware often refuses to run inside VMs to avoid detection. This group checks for registry keys, file paths, and device drivers associated with virtual machines.
Browser manipulation — Does the extension alter browser security settings, change the homepage, or extract private information from locally installed browsers? This directly covers the credential theft and browser hijacking behaviour that Component 1 is designed to detect.
Information stealers — Does the extension collect sensitive information from the computer and the browser? This includes reading cookies, saved passwords, form data, and session tokens — the core malicious behaviour of spyware extensions.
Persistence mechanisms — Does the extension try to install itself to run at startup (autorun), create hidden data streams (Alternate Data Streams in NTFS), or install native executables? Malicious extensions try to survive browser reinstallation.
Network traffic patterns — All outbound network connections are captured in PCAP format. This includes connections to C2 servers, cloud storage services (Dropbox, Google Drive, MediaFire, Mega), and any unusual data transfers that could indicate data exfiltration.
Disabling security features — Does the extension try to turn off browser security warnings, disable system restore, modify firewall policies, or disable OS security features? Malicious extensions often disable defences before executing their payload.
Packing and obfuscation — Does the extension use code packing or obfuscation techniques to hide its true functionality? This is a strong indicator of intent to evade detection — legitimate extensions have no reason to hide their code.
Ransomware behaviour — Does the extension attempt to encrypt files or show ransom messages? While rare in browser extensions, this category is monitored as part of the comprehensive behavioural audit.
Bitcoin/cryptocurrency mining — Does the extension try to install OpenCL or other cryptocurrency mining tools? Browser-based cryptojacking extensions silently use the user's CPU to mine cryptocurrency.

How to use these features in your ML model
Practical steps to load and train on this dataset
▼

The repository provides the 1,098 features already extracted and structured — you do not need to run the sandbox yourself for this dataset. Each extension is one row; each of the 1,098 columns is a binary feature (0 or 1); the label column tells you malicious (1) or benign (0).

import pandas as pd
from sklearn.model_selection import train_test_split
from xgboost import XGBClassifier
from sklearn.metrics import classification_report

# Load the dataset
df = pd.read_csv('dataset.csv') # from the GitHub repo

X = df.drop('label', axis=1) # 1,098 feature columns
y = df['label'] # 0 = benign, 1 = malicious

X_train, X_test, y_train, y_test = train_test_split(
X, y, test_size=0.2, stratify=y, random_state=42
)

# Handle class imbalance with scale_pos_weight
scale = len(y_train[y_train==0]) / len(y_train[y_train==1])

model = XGBClassifier(
scale_pos_weight=scale, # handles 22:990 imbalance
n_estimators=100,
max_depth=6,
random_state=42
)
model.fit(X_train, y_train)

print(classification_report(y_test, model.predict(X_test)))
What to expect
With XGBoost and proper class imbalance handling, you should achieve F1-scores above 85% even with only 22 malicious samples. The 1,098 rich behavioural features compensate for the small malicious class size. When you combine this with the chrome-mal-ids and palant datasets (which add more malicious samples), accuracy will improve further.
